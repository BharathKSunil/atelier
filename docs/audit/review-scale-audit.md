# Atelier — Review Section Audit (Scale, Usability, Feature Gaps)

> Multi-agent audit, June 2026. 31 findings, every finding adversarially verified against the
> code (0 rejected). Scope: is the Review/cull section built for a **local** 20k-image / TB-scale,
> RAW-shooting audience? All recommendations are local-only — no cloud, by constraint.

## 1. Verdict

The review/cull section is **functionally correct and non-destructive, but not yet built for the 20k-image / TB-scale, RAW-shooting audience it targets.** Its real strengths are the right ones for a local tool: persisted thumbnail BLOBs, a metadata-only `/series` payload, `threaded=True` Flask, and a resumable `processed` flag in indexing. The headline risks are all on the human-throughput axis: the cull workflow is **keep-only with no reject/rating vocabulary**, has **no zoom/compare** for the one decision it exists to make (sharpest frame), has **no resume/jump/filter** so multi-day culls restart at burst 1, and **never ingests RAW/HEIC** — the actual deliverable files — silently. The much-hyped "load everything before paint" and "critical perf cliffs" are real code smells but, on a loopback server with tiny integer payloads, are low-to-medium polish, not the cliffs first claimed. **Net: the engine is sound; the cull cockpit is missing the controls a working culler needs.**

## 2. Is it built for scale?

Direct answer: **partly.** The data plane is reasonable; the interaction plane and the first-index are where it breaks down. Ordered by real impact:

1. **First-index is single-threaded with per-image commit** — `atelier/pipeline/index.py:79` is a serial `for row in tqdm(pending)` doing full-res decode (`:91`) + thumbnail (`:100-103`) + DINOv2 (`:106`) + SCRFD/ArcFace (`:107`), with `conn.commit()` *inside* the loop (`:185`). `models.py:70,107` take one PIL image per call — no batch path; no `concurrent.futures`/pool anywhere. On a 20k/TB shoot this is the longest single operation the user faces and pins one core while 7 sit idle. **Local fix:** CPU decode/thumbnail pool feeding a batched DINOv2/SCRFD/ArcFace forward pass; commit every 50–100 rows; keep the `processed` flag for resumability. The ~20k WAL fsyncs are real but sub-ms each — a minor tax next to serial decode, so batch-commit is the easy win and parallel decode+batched inference is the real one. (Effort L)

2. **Hero/lightbox re-decodes the full-res original on every frame change, no preview tier** — `server.py:836-842` `full_image` is a bare `send_file(path)` with **no Cache-Control** (the three thumbnail paths right above at `:808/:819/:833` all set `max-age=86400` — the omission is inconsistent, not principled). `cull.js:113-116` re-points the hero `<img>` at `/image/<id>` on every burst load, every Up/Down (`moveHero` `:287`), and every filmstrip click (`:158`). There is nothing between the ~480px thumbnail and the full original. **Local fix:** add a persisted ~1600px preview tier (decode-once, stored like the thumbnail) for the hero; reserve the original for an explicit zoom/lightbox; set `Cache-Control` on both. Note Flask's `send_file` already emits ETag/Last-Modified, so backtracking gets a 304 — the bytes aren't re-streamed, but the **decode** is re-paid; the preview tier is the actual fix, the header is a loopback-cheap extra. (Effort M)

3. **No next/prev prefetch — every Next pays cold metadata + cold hero** — `step()` (`cull.js:277-280`) just advances `pos` and awaits `load()`; `load()` (`:62-100`) does `/images`+`/picks` then a *third* round-trip `/buckets/for-images`, all on-demand; the hero is only requested at render. No `prefetch`/`preload`/`new Image()` anywhere. On loopback the metadata fetches are sub-ms; the felt cost is the cold full-res hero decode. **Local fix:** after the current burst paints, warm ±1 burst's `/images`+`/picks` and the next hero via an off-screen `Image()`/`link rel=preload` (point it at the new preview endpoint). Bound to ±1 to keep memory flat. (Effort S)

4. **`image_thumb` fallback re-decodes the original every request and never writes back** — `server.py:823-834` opens the original, `draft`+`thumbnail`+re-encodes, and serves it **without persisting** into `images.thumbnail`. Any NULL-thumbnail row (pre-v2, or index-error rows at `index.py:186-189`) re-pays this on every filmstrip paint, forever. The author already knows NULLs exist (dashboard cover filters `thumbnail IS NOT NULL`, `server.py:208`) but the filmstrip path doesn't. **Local fix:** one `UPDATE images SET thumbnail=?` write-back in the fallback branch. Note `im.draft("RGB",(640,640))` (`:826`) lets libjpeg downscale-on-decode, so the JPEG cost is moderate; PNGs pay full decode. (Effort S)

5. **Exports copy serially on the request thread, no LIMIT/progress/cancel** — `export_prints` (`:767-780`), `bucket_export` (`:613-629`), `export_person`/`export_persons` (`:412-455`) each SELECT all paths (no LIMIT) into a list, then `_copy_unique` (`:45-64`) loops `shutil.copy2` on the Werkzeug thread; the request stays open for the whole copy. The deliverable step is the least observable operation in the app. `threaded=True` (`:880`) means the rest of the UI stays live, so this is medium, not high — but a TB bucket export blocks for minutes with no progress or cancel. **Local fix:** reuse the out-of-process `runner.py` pattern (or a worker thread + job-id), stream a per-file counter over the SSE channel already used for run logs, add a cancel button. (Note: the finding wrongly listed `bucket_add_people` here — `server.py:564-585` is a metadata-only `INSERT OR IGNORE`, no copy.) (Effort L)

6. **Quadratic series grouping on timestamp-less frames** — `series.py:52-58` collects every `t is None` item and runs a nested all-pairs `np.dot` per pair. `pipeline/series.py:42` sets `t=None` whenever `exif_time` is False, and `imaging.py:80-82` sets `exif_time=False` for **every** PNG and EXIF-stripped/edited JPEG (not "some" — all). A PNG-heavy or edited-export import makes this set large and the phase appears to hang. **Local fix:** reuse the block-tiled cosine pattern already in `cluster.py:132-162`, or cluster the timestamp-less embeddings like faces. (Effort M)

7. **Whole-table embedding loads in cluster/series/score** — `cluster.py:37,45-46` np.stacks every 512-d face embedding (plus a normalized copy); `series.py:25-45` loads all 384-d globals; `score.py:55,86-89` materializes all faces and all processed images — and `.fetchall()` there also pulls the 360px thumbnail BLOB per row (~40KB × N), a larger spike than the embeddings. Fine at 20k (~100–130MB); a true hundreds-of-thousands-of-faces project risks swap/OOM on a laptop. **Local fix:** chunk/memmap the base reads; mini-batch/approximate clustering for the large case. (Effort M)

8. **Load-all-series-before-paint** — `cull.js:28-33` paginates `/series` to exhaustion before `load()` at `:52` (spec-locked at `review-cull.feature:32`). **This is the most over-hyped item:** `/series` returns 4 small integer columns per row (`server.py:638-644`), only `frame_count>1` series (so far fewer rows than images), over loopback. Even 10k bursts is a few hundred KB and ~50 sub-ms calls — sub-second, not "seconds-to-minutes." Still worth fixing because it **blocks any lazy jump-to-undecided design** (see §3). **Local fix:** render after page 1, background-page the rest; expose a server `total` so the `series.length`-based counter at `cull.js:109-110` still works. (Effort S — low urgency)

## 3. Review-section usability

Ordered by photographer-pain:

1. **No reject/X/cut flag — culling is keep-only** — `cull.js:314-344` binds only star (Space/S), buckets (1-9), criterion (g/c/a). There is no photo reject; `pick_type` is whitelisted to `group/aesthetic/candid`+`print` (`config.py:69`, validated `server.py:701`), and the `picks` table (`db.py:131-138`) has no reject column. The `reject` at `server.py:394` is **face-cluster** cleanup, not photo rejection. Sharp edge: **`x`/`X` is bound to `step(1)` "next burst"** (`cull.js:321`) — the universal Lightroom/Photo-Mechanic reject key silently advances instead. Real culling at 20k is subtractive (flag throwaways fast, print survivors); keep-only forces positively touching every keeper. Workaround today: co-opt a 1-9 bucket as "reject," but there's no filter to hide those bursts. **Fix:** add a `reject` pick_type (new key, since `x` is taken), a Review filter to skip rejected bursts, and a copy-to-`_rejects` export (never delete originals). (Effort M)

2. **No zoom-to-100% — can't verify the sharpness the tool exists to judge** — hero is `object-fit:contain` (`style.css:425-428`), lightbox `object-fit:contain` (`style.css:1423-1426`), and there is **no** wheel/pinch/scale/transform handler anywhere in `cull.js`/`faces.js`. The full-res original *is* served (`server.py:837`) but always fit-to-screen. On a 24MP+ frame, eyes-open/critical-focus differences are invisible. This is worse than "no synchronized zoom" — there's no zoom at all. **Fix:** click/scroll zoom+pan on the already-fetched hero (CSS transform), or a 100%-crop toggle centered on the detected face box (face box data already exists, `faces.js:72`). (Effort M)

3. **No compare / 2-up / N-up** — the stage renders exactly one hero (`cull.js:113-116`); Up/Down swap the single hero (`moveHero` `:281-290`); lightbox is single-image prev/next (`faces.js:12-25`). The single most common burst decision — eyes-open A vs eyes-open B — requires flicking back and forth from memory. **Fix:** pin-current-hero key → show pinned vs current side-by-side using already-fetched images, with synchronized zoom. Pure frontend over `/image`. (Effort L)

4. **No resume / no "reviewed" state** — `pos` is a module-level var (`cull.js:14`), Math.min-clamped on mount (`:51`), never persisted; `unmountReview` (`:55-60`) saves nothing, so a **full page reload** resets to burst 1. No `reviewed_at`/`seen` column in any schema version (`db.py`). (It survives an in-app tab switch — the module isn't re-evaluated — so only reload truly loses it.) A multi-day 20k cull returns to burst 1 of 3,000 with no memory of coverage; a burst reviewed-and-rejected is indistinguishable from one never opened. `is_print` stars do persist, so starred bursts carry a partial trace. **Fix:** per-series `reviewed_at` (local migration) set on advance-past/mark-done; persist last position in localStorage; checkmark reviewed bursts; resume at first unreviewed. (Effort M)

5. **Linear stepping only — no jump, no filter to undecided** — navigation is exclusively `step(±1)` mod `series.length` (`cull.js:277-279`); the Review bar (`index.html:36-46`) has no jump field, slider, or "only no-pick / only unstarred / only large bursts" filter. The `/series` payload carries `frame_count`/`best_score` (`server.py:639-641`) but `cull.js` reads only `.id` — the signals are fetched and ignored. Partial mitigation the finding omits: bursts come `ORDER BY frame_count DESC` (`server.py:642`), so the biggest surface first, and wrap-around caps worst-case at ~length/2, not the full list. Still, "show me the 40 bursts with no keeper" — the exact wedding triage — is impossible. **Fix:** a burst jump/index control + client-side filters over the loaded array, plus a per-series `has_print` flag so "only undecided" needs no per-burst frame load. (Effort M)

6. **Single-frame bursts never appear in Review** — `/series` hard-filters `frame_count>1` in COUNT and SELECT (`server.py:637,641`; spec-locked `review-cull.feature:17-23`). Singletons are real DB rows (`pipeline/series.py:50-61` maps every image to a series) with a computed `print_score`, but Review is the **only** surface with star/bucket actions, so the ceremony kiss, the single portrait, the ring shot can never be starred to the print list. The "hunt via People" escape hatch is weaker than implied: People is faces-only with no star action, so a **faceless** singleton is unreachable entirely. **Fix:** a paged "singles" review mode (lazy, like People) with the same star/bucket/zoom shortcuts, or a toggle to include singletons. (Effort L)

7. **No undo for any review action** — `star`/`setCriterion`/`toggleBucket`/`rangeStar` all commit immediately with only an auto-dismiss toast; `onKey` (`:314-344`) has no Ctrl/Cmd+Z. Single actions are self-reversible (star and bucket are toggles — same key flips back), so the acute risk is concentrated in **`rangeStar`/`star_many`** (`cull.js:219-253`): a stray shift+click stars dozens of frames with **no bulk inverse**. **Fix (minimum):** give the range-star result an Undo affordance — which requires extending `toast()` (`api.js:47-65`) to accept an action — or a bulk-unstar. A full last-N Ctrl+Z stack is a nice-to-have. (Effort M, or S for the range-star Undo alone)

8. **No EXIF / shot-data surfacing for tie-breaks** — `read_metadata` (`imaging.py:46-82`) captures only `taken_at`/camera/orientation/dims; ISO/aperture/shutter/focal/lens are **never read or stored** (no columns in `db.py`). And even the captured date/camera/dims are shown **nowhere** in Review — `series_images` (`server.py:652-654`) doesn't even send them to the cull client. When two frames look equal, the data a photographer normally decides on (1/200 vs 1/60, f/1.4 vs f/4) is absent. **Fix:** capture full EXIF at index; surface a compact shot-data overlay on the hero and in the lightbox caption. (Effort M)

9. **No frame sort/filter inside a burst; quality signals hidden** — `displayFrames` is frozen once in `load()` (`cull.js:88`) and never re-sortable; the endpoint returns `global_sharpness`/`exposure_score` per frame (`server.py:652`) but `cull.js` never reads them — no per-frame number, no "sharpest first." In a 30-frame burst you eyeball tiny thumbnails in fixed order. **Fix:** show sharpness/exposure on hover + a client-side sort toggle (no re-fetch; the data already arrives). (Effort S)

10. **Shortcuts under-discoverable; "X skip" mislabeled** — the full set lives only in a static help-fab (`index.html:50-53`) + three tooltips; **shift+click range-star** and **click-stage-to-lightbox** are documented nowhere, criterion *semantics* (`cull.js:5-9`) are never rendered, and "X skip" (`index.html:51`) actually just means next (`step(1)`), implying a defer/skip state that doesn't exist. **Fix:** a `?`-toggled legend overlay covering range-star + stage-click + criterion meaning; relabel "skip"→"next." (Effort S)

## 4. Missing features / tweaks (vs the stated audience)

### Ingest (RAW/HEIC)
- **No RAW/HEIC decode — the deliverable files are silently ignored.** `IMG_EXTS = {".jpg",".jpeg",".png"}` (`imaging.py:10`); `iter_images` (`:18-21`) yields only those, so a CR2/NEF/ARW/DNG/HEIC card dump indexes the in-camera JPEGs or **nothing** — and the run report (`index.py:191-194`) counts only indexed/faces/errors, so a skipped RAW leaves **no trace**. No `rawpy`/`pillow_heif` anywhere (only a `pillow-heif` note in `REQUIREMENTS.md:214`). This is the single biggest adoption blocker for wedding/event shooters. **Why it matters:** their 20k-image dump is mostly RAW; pointing Atelier at it culls the JPEG sidecars or fails silently. **Local fix:** `rawpy` (libraw) for RAW + `pillow_heif` for HEIC, registered before `iter_images`; index a downscaled embedded preview/half-size decode for scoring while keeping the RAW on disk as the export "original." **Ship the cheap half independently:** a per-run "scanned N / indexed M / skipped K (unsupported)" counter so a RAW-only folder is distinguishable from a broken run. **(Effort: XL for decode; S for the skip counter)**

### Rating & triage vocabulary
- **No 1–5 stars or color labels — selection is one boolean.** `PICK_TYPES` fixed to `group/aesthetic/candid` (`config.py:69`); `star()` toggles a single `print` row (`server.py:720-736`); no rating/label column (`db.py:131-138`). Buckets (keys 1-9) give a *manual* multi-tier workaround, so "tiered delivery is impossible" overstates it — the real gaps are no numeric-rating **ordering** and no interchange with the rating model Lightroom/C1 expect. **Why it matters:** pros deliver in tiers (5★ album vs 3★ maybe). **Local fix:** integer rating (0–5) + color-label enum in SQLite; rating shortcuts in Review (move buckets to a modifier since 1-9 are taken); let print/export filter by tier. **(Effort M)**
- **No reject flag** — see §3.1. **(Effort M)**

### Compare / inspect
- **No 100% zoom** (§3.2, M) and **no 2-up/N-up compare** (§3.3, L) — the two inspection tools the headline "pick the best frame" claim depends on. **No EXIF surfacing** (§3.8, M). All pure-local frontend/Pillow work over the existing `/image` endpoint and `getexif`.

### Delivery / export
- **No XMP/IPTC sidecar writing — decisions don't round-trip.** Every export is `shutil.copy2` only (`_copy_unique` `server.py:45-64`, backing all `export_*`); no XMP/exiftool/piexif anywhere. **Why it matters:** the photographer culls in Atelier, re-imports bare JPEGs into Lightroom with no ratings/picks/labels — selection is redone. Caveat: grouping **does** survive as exported folder structure (one folder per person/bucket/print), so "every decision is lost" overstates — it's the per-image rating/label/pick flag that's lost. **Local fix:** write per-image `.xmp` sidecars (`xmp:Rating`, `xmp:Label`, pick flag) next to each copied original via a small template or `python-xmp-toolkit`; optionally embed with `piexif`. **(Effort L)**
- **Print-list export has no folder picker — delivery footgun.** `prints.js:107` posts empty `{}`; `export_prints` defaults dest to `./print_exports/<slug>` relative to **server CWD** (`server.py:770`), unlike Buckets/People which call `/api/fs/choose` first (`buckets.js:208-226`). Worst case: if the server was launched outside an allowed root (`_allowed_roots` `:94-103`), `_safe_dest` returns None and the export **hard-fails 400** with no picker to recover. The single most important export is the only one without a picker. **Local fix:** wire `/api/fs/choose` into the Export-all button, require explicit dest, default the picker to `/Volumes` for delivery drives. **(Effort S)**
- **Exports flatten folder structure; collisions silently renamed.** `_copy_unique` (`:53-58`) copies every source into one flat dest by basename, suffixing collisions `_1/_2`. Two bodies' `IMG_0001.JPG` on the same day collide; the delivered set no longer mirrors the catalog/Day/Cam convention. The rename is deliberate collision-safety (docstring `:46-47`), not accidental overwrite — but it's unsurfaced. **Local fix:** a "preserve source folder structure" option recreating the relative subtree under dest, anchored at `args.photos` (full path is already stored, `index.py:69`). **(Effort M)**
- **No server-side zip despite UI promising it.** No `zipfile`/`make_archive` anywhere; `buckets.js:116`/`people.js:68` say "zip and share." The prose is defensibly "we give you a folder, you zip it," and in-process zipping a multi-GB selection transiently doubles disk — so this is **low/polish**, not a defect. **Local fix:** optional `shutil.make_archive` to the chosen dest. **(Effort M, low priority)**

### Other
- **No sort/filter anywhere except People name-search.** Hard-coded sorts: Review `frame_count DESC` (`server.py:642`), prints `series_id` (`:762`), buckets `added_at DESC` (`:608`), people `cnt DESC` (`:338`). `taken_at`/`camera` are **captured but never selected into any response or used in any WHERE/ORDER BY** — the chronological axis a wedding cull lives on is dead data. (Correction to the raw claim: `/persons` *does* honor a `q=` name filter, wired in `people.js`, so it's not "zero filters" — it's no date/camera/rating/has-faces filter and no sort control.) **Why it matters:** "cull chronologically / jump to the reception / isolate Cam B" is impossible. **Local fix:** server-side sort+filter params on existing paginated endpoints + a filter bar; `taken_at`/`camera` need a trivial new index (`db.py` indexes none of them). **(Effort L)**
- **No ICC/color management.** `.convert("RGB")` with no profile (`imaging.py:28`); no `ImageCms`. Scoped impact: the colorfulness term (`quality.py:28-36`, weight 0.40) feeds **only** `aesthetic_score`, i.e. the opt-in "Striking" criterion — the `group`/`candid` print criteria use sharpness/exposure/eyes and are **unaffected**, so "systematically biased across the whole shoot" overstates it. The hero serves **original bytes with ICC intact** (`server.py:837`) so a calibrated browser renders it correctly; only the **thumbnails** are saved untagged (`index.py:103,140`, `server.py:830`) and look desaturated. **Local fix:** apply the embedded ICC via `ImageCms` before scoring; pass `icc_profile=` when saving thumbnails. **(Effort M, low-medium severity)**
- **Merge-into modal loads every person.** `people.js:342-353` paginates all persons to exhaustion (each with a thumbnail) then filters client-side, though `/persons?q=` exists (`server.py:320-341`). Fires on opening merge, not on People paint. Caveat: `q=` filters only `display_name`, so unnamed clusters (the common case) need the server search extended to match by id before a pure `q=` rewrite. **Local fix:** drive the picker off debounced `/persons?q=`. **(Effort S)**

## 5. Prioritized roadmap

| Priority | Item | Category | Effort | Why |
|---|---|---|---|---|
| P0 | RAW/HEIC ingest (+ skip counter) | Ingest | XL (S for counter) | Target audience's deliverables are RAW/HEIC; silent zero-index is the #1 adoption blocker. Ship the skip-counter first. |
| P0 | Reject/X flag + hide-rejected filter | Triage / usability | M | Subtractive culling is how 20k frames get culled; keep-only forces touching every keeper. `x` is even mis-bound to "next". |
| P0 | Zoom-to-100% on hero/lightbox | Compare/inspect | M | The tool's core job is picking the sharpest frame; no zoom = picking blind on 24MP files. |
| P0 | Print-list export folder picker | Delivery | S | The headline deliverable lands in a CWD-relative path or 400-fails; trivial frontend fix to an existing picker. |
| P1 | Resume + per-burst "reviewed" state | Usability | M | Multi-day culls restart at burst 1 with no coverage memory — doubles the most expensive human step. |
| P1 | Jump-to-burst + undecided/unstarred filters | Usability | M | Targeted triage ("the 40 bursts with no keeper") is impossible under linear stepping. |
| P1 | Next/prev prefetch + hero preview tier | Scale-perf | S + M | Kills the per-Next stall — the friction multiplier across thousands of bursts on slow/external disk. |
| P1 | XMP sidecar writing on export | Delivery | L | Without it, every per-image rating/pick/label is lost re-importing to Lightroom/C1. |
| P1 | 2-up/N-up compare | Compare/inspect | L | The eyes-open-A-vs-B decision is the most common burst call and currently relies on memory. |
| P1 | Singles review mode | Usability | L | Most single portraits/details/ring shots can never be starred to the print list. |
| P1 | Batched/parallel first-index | Scale-perf | L | Longest single operation on a 20k/TB shoot; serial decode wastes 7 idle cores. |
| P2 | 1–5 rating + color labels | Triage | M | Tiered delivery; numeric ordering and downstream-tool interchange (buckets are a clumsy stand-in today). |
| P2 | Sort/filter (date/camera/rating) + EXIF surfacing | Usability / feature | L + M | Chronological/camera slicing a wedding cull needs; `taken_at`/`camera` already captured but dead. |

Lower-priority polish (P2/P3, noted but off the top-12): `image_thumb` write-back (S), export structure-preservation (M), undo stack / range-star Undo (M/S), shortcut legend overlay + "skip"→"next" relabel (S), in-burst frame sort + sharpness overlay (S), merge-modal server-side search (S), ICC for thumbnails+scoring (M), server-side zip (M), quadratic series grouping (M), streamed/cancelable exports (L), chunked embedding loads (M). The load-all-series-before-paint item (S) is worth doing **only** as the enabler for lazy jump-to-undecided — on its own it's sub-second on loopback, not a cliff.
